# 实时翻译页面二次改造需求理解

日期：2026-05-15

## 背景

当前页面已经能完成会议实时翻译、中文回复生成、英文语音播放，但在长时间学校发布会等真实场景中暴露出四类问题：

1. 阅读区域不合理：原文英文区域经常为空，却占据大量空间；中文翻译区域偏小，长内容需要频繁滚动。
2. 控件干扰阅读：开关、按钮、播放器等控制项长期占据页面上方和正文区域，影响聚焦翻译内容。
3. 录音控制不够像会议工具：希望用蓝牙麦克风或按键实现类似 PTT 的开始/停止控制。
4. 成本保护不足：关闭中文译声目前只是浏览器静音，不等于停止生成译声音频；长时间忘记停止会持续消耗费用。

## 当前实现判断

### 原文英文看不到

不是刻意设计成“不看原文”。更准确地说：当前线上入口使用的是新 POC 页面 `public/realtime-translation-poc.html`，这个页面没有复用原始页面 `public/trans.html` 里已经接入的转写链路。

原始 `public/trans.html` 已经包含多套转写接入：

- `/api-proxy/api/openai/whisper`，通过后端代理调用 OpenAI Audio Transcriptions。
- `streamTranscribe()`，把 PCM/WAV 分段送到 Whisper，边录边更新文本。
- Deepgram 实时转写逻辑，累积最终结果并触发文字翻译。
- BLE 麦克风录音与 ADPCM/PCM/WAV 转换逻辑。
- Deepgram + Whisper 的转写结果合并逻辑。

当前新 POC 页的 realtime translation session 只配置了：

```json
{
  "session": {
    "model": "gpt-realtime-translate",
    "audio": {
      "output": {
        "language": "zh"
      }
    }
  }
}
```

这条 realtime translation POC 链路没有复用旧页面的 Whisper/Deepgram/BLE 转写能力，也没有在 realtime session 中额外配置 `audio.input.transcription`。因此页面上的“英文原文”格子在实时译声模式里可能没有稳定内容。

后续改造应优先复用/迁移原始 `trans.html` 中已经验证过的转写链路，而不是重新发明一套完全独立的录音转写流程。

### 关闭“听中文译声”是否省钱

当前不是。现在开关只在浏览器端静音并暂停 `remoteAudio`。OpenAI 侧 realtime translation 仍然会生成译声音频并返回到浏览器，因此它解决的是“不要听到声音”，不是“不要产生声音成本”。

## 推荐改造方向

### 1. 页面阅读模式

把页面从固定左右两个文本框，改成“主阅读区 + 可折叠辅助区”：

- 默认主阅读区显示中文翻译，占页面 70%-85% 高度。
- 英文原文只在启用“显示/生成原文”时出现，否则折叠成小标签。
- 回复区默认折叠，点击“回复”按钮或快捷键时展开。
- 控件放到右侧悬浮按钮或抽屉，不长期占正文空间。

### 2. 模式选择

增加清晰的翻译模式选择：

- `实时译声模式`：继续使用 `gpt-realtime-translate`，低延迟、自然、有中文译声，但成本较高。
- `省钱文字模式`：不使用 realtime translated speech；走转写 + 文本翻译链路，只输出文字，按段更新，成本更低但延迟更高。
- `文字 + 原文模式`：在省钱模式基础上同时显示英文原文和中文翻译，适合复盘和精读。

### 3. 长会成本保护

增加会话上限与提醒：

- 默认单次最多 60 分钟。
- 剩余 5 分钟、3 分钟、1 分钟显示大号倒计时提醒。
- 到时间自动停止，不自动续费。
- 提供“延长 15 分钟”按钮，但必须用户主动点击。
- 页面失焦或静音长时间无输入时，可增加空闲停止策略。

### 4. 蓝牙麦克风 / PTT 控制

Web 页面无法保证读取所有蓝牙耳机实体按键事件，因为很多蓝牙按键会被系统或浏览器拦截。可行方案分三层：

- 第一层：键盘快捷键，例如空格或 `M` 键切换开始/停止。
- 第二层：蓝牙键盘/遥控器映射为键盘事件，实现按一下开始、再按一下停止。
- 第三层：如果某个蓝牙麦克风能在浏览器产生 Media Session 或 HID 事件，再做设备专用适配。

## 建议实施顺序

1. 先做 UI 阅读体验和控制抽屉，立刻改善会议阅读。
2. 再做会话上限、倒计时提醒、自动停止，先堵住忘关造成的成本风险。
3. 然后做模式选择：实时译声模式 vs 省钱文字模式。
4. 最后做 PTT/蓝牙按键适配，因为设备差异最大，需要实际测试。

## 待确认

最关键的产品选择：默认打开页面时，应默认进入哪种模式？

- 推荐默认：`省钱文字模式`，需要中文译声时再切换到 `实时译声模式`。
- 备选默认：继续默认 `实时译声模式`，但加 60 分钟硬停止和明显成本提示。
