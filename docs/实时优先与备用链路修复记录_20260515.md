# 实时优先与备用链路修复记录

日期：2026-05-15

## 背景

线上省钱文字模式出现 `Audio file might be corrupted or unsupported`。根因是浏览器 `MediaRecorder.start(timeslice)` 生成的连续 WebM 小片段，后续片段不一定是完整可独立解码的音频文件，转写接口会拒绝这些片段。

## 本次调整

1. 新页面默认回到实时译声模式，优先体验 `gpt-realtime-translate`。
2. 省钱文字模式作为备用 fallback 保留，不生成中文译声。
3. 省钱文字模式改成每段独立 `start/stop`，每段生成完整音频文件后再提交转写。
4. 对 `corrupted`、`unsupported`、`could not be decoded` 等可恢复音频错误，Cloudflare Function、本地 `api-proxy`、前端都会跳过坏片段并继续录音。
5. 新页面增加话筒选择和刷新按钮，可以选择系统识别到的默认话筒、蓝牙话筒或其它输入设备。
6. PTT 仍保留为空格键切换开始/停止；如果蓝牙遥控器能映射为空格，也可以触发页面工作。

## BLE 与旧链路保留策略

- 旧页面 `public/trans.html` 中已有的 Whisper、Deepgram、BLE 接入逻辑本次不移除。
- 当前新页面先通过浏览器标准 `enumerateDevices/getUserMedia(deviceId)` 支持系统级蓝牙麦克风选择。
- 专用 BLE 按键或自定义 BLE 协议后续可从旧页面迁移为独立适配层，用来触发新页面的开始/停止或 PTT 状态。
- Whisper、Deepgram 分段转写逻辑保留为次要 fallback，不作为当前主体验。

## 验证点

- 前端静态测试覆盖：默认实时模式、备用省钱模式、控制抽屉、主要阅读区、英文原文开关、话筒选择。
- Cloudflare Function 测试覆盖：坏音频片段返回 `skipped: true`，不再 500 中断。
- 本地 `api-proxy` 测试覆盖：同样跳过可恢复坏音频片段。
